1 Technical Architecture

✦

The platform is built on a cloud-native, event-driven architecture hosted on Microsoft Azure. The core design principle is separation of concerns — raw sensor data flows through a dedicated ingestion layer before being normalised and enriched in a structured processing pipeline, ensuring data quality is enforced at the earliest possible stage rather than at query time.

Core technology stack

✦
  • Ingestion Azure IoT Hub — manages 1,200+ sensor connections with guaranteed delivery and schema validation at the edge
  • Streaming Azure Event Hubs — processes 50,000+ events/second with millisecond latency for real-time risk signal processing
  • Pipeline Databricks + Delta Lake — unified analytics engine with ACID-compliant lakehouse storage for sensor and ERP data
  • Orchestration Azure Data Factory — manages ETL pipelines with dependency resolution, retry logic, and lineage tracking
  • AI / ML Azure ML Studio + MLflow — model training, experiment tracking, and automated retraining on drift detection
  • API Layer Azure API Management — rate-limited, OAuth2-secured endpoints for dashboard and third-party integrations
  • Frontend Trmeric Platform — React-based analytics canvas with role-specific executive and operational views
  • Compliance Microsoft Purview — automated data classification, lineage, and audit trail for HIPAA and AS/NZS 4801 requirements

The Salesforce CRM and Google Analytics connectors will use OAuth2 device-flow authentication with token refresh handled by the API Management gateway. Field mapping tables have been pre-defined for 47 CRM attributes and 23 GA4 event dimensions. A schema registry in Delta Lake will version all field mappings, enabling upstream schema changes to be detected and surfaced as breaking-change alerts before they propagate into the analytics layer.
✦

Downstream, all processed data is served via a GraphQL API layer that enforces row-level security based on the user's site access profile. Executive users receive aggregated, cross-site views, while site safety managers are restricted to their assigned geographic cluster. This access model was validated in a security design review with BHP's CISO team on Jan 28, 2025.

4 Key Decisions & Assumptions

✦

The following architectural decisions are locked and should not be revisited without a formal change request to the steering committee. Each carries a date and owner to maintain an auditable record of the solution rationale.

D1
Azure over AWS for primary cloud platform
BHP's enterprise agreement with Microsoft provides a 28% discount on Azure consumption. Existing BHP IT team has Azure skills. Databricks runs natively on both; no re-work needed.
Jan 15, 2025
Kevin V.
D2
Trmeric as primary analytics and AI orchestration layer
Trmeric's demand-resource management modules replace a planned custom build. Estimated to save 6 weeks of development time and provide an enterprise-tested UI for the safety dashboard.
Jan 18, 2025
Arjun M.
D3
Batch export pattern for SCADA integration (Phase 1)
Real-time streaming from OSIsoft PI is not approved by BHP IT Security for Phase 1. 15-minute batch export via data diode is the approved pattern. Real-time to be re-evaluated in Phase 2.
Jan 28, 2025
